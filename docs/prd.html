01

Product definition

Product: the reliability plane — four declared policy objects (Placement, Failover, Release, SLO) enforced by MCM and the router, observed in a single operate console, operated by agents under a closed allowlist, and sold as an SLA ladder (Managed 99.9 / Declared 99.95 / Operated 99.99). Available to two buyers from day one: teams evaluating Baseten for dedicated inference (the console is their evaluation harness — agentic onboarding, real SLO baselines, certified migration when the evidence says so) and existing customers running Baseten inside a multicloud portfolio (the console is their daily reliability surface, and the ledger their case for consolidating traffic).

The trust machinery is the observability contract: machine-readable failure reasons, capacity-wait states, metrics completeness timestamps, and published rate-limit budgets — the console never lies, so the policies can be believed.

The adoption machinery is defaults-with-override: every policy object ships pre-filled with what Baseten's operators would do anyway; declaring is one edit, not a config project.

The retention machinery is accumulated policy: declared SLOs, placement rules, release gates, and incident history compound switching costs the way raw endpoints never do.

The acquisition machinery is the migration play: read-only monitoring of the clouds customers already run on (the wedge nobody is suspicious of), a route ledger that accumulates cross-provider SLO and cost evidence, and certified migration offered exactly when the evidence — and the customer — are ready. Control features stay Baseten-side: observe → migrate → declare → operate.

Supporting metrics: % dedicated workloads under ≥1 declared policy; operate-console WAU; MTTR p50/p95 on agent-covered incidents; goodput at declared SLO; zero-downtime deploys as % of promotions; preemption events honored per policy; external pools attached and traffic won back on measured evidence; SLA-ladder ACV.

Goals and non-goals

Goals (1) The observability contract closed and versioned; (2) the operate console as the daily surface for dedicated accounts; (3) all four policy objects GA with defaults-with-override; (4) right-of-way for compliance-bound workloads live on sensitive capacity; (5) measurable MTTR decline via agent remediation; (6) the SLA ladder monetizing declared reliability.

Non-goals Engine work (BIS-LLM performance is its own roadmap); multi-cloud abstraction resale; a generic APM product (export to Datadog stays first-class); agent autonomy beyond the closed allowlist; per-token pricing changes; replacing MCM's scheduler — the plane declares intent, MCM keeps enforcing it.

07

Always-on tracks

Honest numbers. Every metric on any surface is measured, never estimated; empty windows are labeled lag or no-traffic, never faked; savings are reported against declared baselines, never invoiced from counterfactuals. This rule is inherited by the reference implementation and is non-negotiable — a reliability product that fabricates a number once is dead.
Agent governance. The allowlist is versioned and auditable; learning tunes cadence and thresholds, never expands authority; every agent action is replayable end-to-end; a public MTTR benchmark methodology invites external rerun.
Contract stability. The observability contract (F0.1–F0.3) is versioned API surface with deprecation policy — agents and consoles are customers too (persona P4), and breaking them silently is a reliability incident.

The MVP: the reference implementation

Phase 0 defines the product surface; the reference implementation makes it tangible before a single platform engineer is committed. It is a static, Vercel-deployable operate console that replays real recorded evidence — the measured MTTR episodes, the chaos drills, the friction-log failure taxonomy — against the four policy YAMLs, built from the ai-native-pipeline base (release engine, incident agent, measured-cost ledger, policy schemas). Its architecture is the story in one picture:

Five commitments:

1 · One screen (F0.4): SLO hero, pool grid, placement feed with reasons, release track, incident timeline — the operate.html design contract, glanceable in 5 seconds.
2 · Four policies as YAML (F1.1–F1.5): slo-, release-, placement-, failover-policy.yaml with defaults-with-override semantics, viewable and editable in the console with simulated enforcement.
3 · The agent loop replayed (F2.1): committed incident episodes drive the timeline — quarantine → probe → reinstate with the MTTR stopwatch; a drill button re-runs the loop live in simulation.
4 · Honest numbers throughout: every figure traces to a committed CSV/JSONL or is labeled SIMULATED; the silent-null rule is implemented, not just specified.
5 · Shareable by default: clean public repo, MIT, README mapping every panel to the JD's four outcome areas and to F-numbers in this PRD.
10

Open questions

MCM's policy readiness — how much of the scheduler already keys on per-customer constraints vs needs refactoring to honor declared intent? Phase 0 discovery with the MCM team decides Phase 1 sequencing.

SLA actuarial basis — how many quarters of drill evidence does finance/legal need before selling 99.99% with capacity terms included, and at what reserved-floor threshold?

Preemption economics — what compensation (credits, priority refund) makes right-of-way preemption acceptable to the preempted party without gaming?

Agent liability — what does the Operated-tier contract say about an agent action that worsens an incident, and does shadow-mode history change the answer?

Chains parity cost — rolling deployments are unsupported for Chains today; is that architectural or just unbuilt? It gates ReleasePolicy's "no excluded products" claim.

Enterprise-tier validation — the friction log is n=1 on a non-enterprise workspace; which walls (GPU gating, replica caps, metrics budgets) exist at enterprise tier? Phase 0 telemetry answers this with data.
