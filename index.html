01 · WE GOT OUR HANDS DIRTY

Six dedicated-deploy attempts on Baseten, logged first-hand. Four patterns kept repeating.

You can't tell slow from stuck from doomed

A first deploy sat 30 minutes in silent scheduling limbo, then died as INACTIVE — no queue position, no "waiting for capacity," no reason. GPU families turn out to be org-gated, but you learn it only after packaging and uploading.

friction #6 · #13

Failures you can't triage

BUILD_FAILED with zero API-visible logs. A fixed-RAM SKU that OOM-crash-loops an 8B model with no pre-flight warning. A green build that dies at import time. Three attempts, three distinct failure layers — none diagnosable programmatically.

friction #7 · #12 · #14

State that lies

A failed first deploy kept the production pointer — the healthy fix served 500s until a manual promote. Metrics arrive as silent nulls indistinguishable from "never served," and the metrics endpoint rate-limits a single dashboard user.

friction #15 · #18 · #19 · #10

The fast path is hidden

One opt-in config stanza (BDN weight caching) cut cold start from 360s to 148s — measured. Nothing at push time suggests it. The platform's best behavior is opt-in and undiscoverable exactly where a new customer would quit.
